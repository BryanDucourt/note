# FeatSched motivation

- 将 benchmark 工作流从 TailBench 硬依赖转向 PTS。
  - 增加 PTS 安装和运行文档，记录 `phoronix-test-suite`、`php-cli`、`search/info/install/batch-run`、`result-file-to-json/result-file-to-csv` 等流程。
  - 在 `configs/workloads.yaml` 中加入 PTS CPU、memory/cache、scheduler-sensitive、web/service 候选 workload。
  - 新增 `analysis/parse_phoronix_result.py`，把 PTS `result-file-to-json` 输出转换为 schema v2 `run_summary.csv`。
  - 保留 TailBench 为 optional/legacy，不作为默认依赖。

- 改进 PTS sanity 与导出流程。
  - 新增 `scripts/export_pts_result.sh`，默认 dry-run，只有 `--execute` 时才调用 PTS export。
  - 支持 JSON 导出和可选 CSV 导出，并检查目标 JSON 是否实际生成。
  - 文档中明确：PTS JSON 是 run-level objective，不是 window-level `app_metrics.csv`。

- 改进 benchmark 与 collector 的 cgroup 配合。
  - `runners/run_benchmark.py` 支持把 benchmark 子进程放入已有 cgroup。
  - 避免使用 `preexec_fn` 直接迁移 cgroup，改成 trampoline：父进程先把子进程 pid 写入 `cgroup.procs`，成功后再让子进程 exec workload。
  - 增加输出目录可写性检查，避免 root-owned `run_meta.json/stdout/stderr/exit_code` 导致 traceback。
  - `scripts/run_collectors_concurrent.sh` 支持把绝对 cgroup path 规范化为 `perf stat -G` 需要的相对 selector。
  - 修复 sudo collector 创建 root-owned 输出目录的问题：通过 `SUDO_UID/SUDO_GID` 把新建目录和 collector artifacts 归还给原始用户。

- 记录并提交了一次真实 PTS paired run。
  - 保留最后一次完整结果：`runs/pts-c-ray-once-paired-20260609-02`。
  - 包含 benchmark `run_meta.json`、`run_summary.csv`、`pts_result.json`，以及 collector 的 `window_features.csv`、PMU/cgroup/scheduler raw CSV。
  - 其他旧的失败/无效 run 没有进入 Git。

- 分析上次 paired run 中 `window_features.csv` 的 NaN 原因。
  - 结论：NaN 主要来自 collector 固定采 240s，而 PTS c-ray 实际有效运行约 208.5s。
  - 前约 8.3s 是 benchmark 真正开始前的空窗，后约 14s 是 benchmark 结束后的空窗。
  - cgroup CPU 数据基本完整；PMU/scheduler 在 idle 或无事件窗口中出现 NaN 是合理现象。
  - `app_*` 全部 NaN，因为该 run 只有 PTS run-level objective，没有 window-level app metrics。

- 新增 paired run 一致性检查。
  - 新增 `analysis/check_paired_run.py`。
  - 检查 `<run-root>/benchmark` 和 `<run-root>/collectors` 下的 `run_meta.json`、`run_summary.csv`、`window_features.csv` 是否成对一致。
  - 校验 `run_id`、`policy`、`workload`，支持显式 `--allow-workload-alias`。
  - 没有 window-level app metrics 时，只接受明确 notes：
    - `PTS_run_level_objective_only`
    - `run_level_objective_no_window_app_metrics`
    - 历史 `collector_smoke_no_app_metrics`，但会 warning。

- 新增 window feature 聚合。
  - 新增 `analysis/aggregate_window_features.py`。
  - 输入 `collectors/window_features.csv`，输出 `collectors/run_feature_summary.csv`。
  - 对每个数值特征输出 all/active 两组统计：
    `mean`、`median`、`p95`、`p99`、`std`、`non_nan_count`。
  - 默认 active 窗口定义为：
    `cpu_util >= 0.5 OR cycles 有数值 OR instructions 有数值`。
  - 输出 active window 数量、持续时间、比例、首尾 active timestamp，方便后续把 run-level PTS objective 和 active-window features 对齐。

- 改善重复实验的人机流程。
  - 新增 `scripts/prepare_experiment_run.sh`。
  - 每次实验只需生成一次共享 env：
    `runs/<RUN_ID>/run_env.sh` 和 `runs/current_run_env.sh`。
  - collector terminal 和 benchmark terminal 都 `source runs/current_run_env.sh`，避免手动同步 `RUN_ID`、`RUN_ROOT`、`COLLECT_OUT`、`BENCH_OUT`、`CGROUP_PATH`。
  - 生成的 env 提供：
    - `featsched_show_run`
    - `featsched_collectors_plan`
    - `featsched_run_collectors`
    - `featsched_benchmark_plan`
    - `featsched_run_benchmark`

- 在仓库根目录新增中文实验提醒。
  - 新增 `tips.md`，记录重启后、新开 terminal、每次实验前需要执行的检查步骤。
  - 重点提醒：
    - benchmark 不要用 `sudo`；
    - benchmark terminal 需要先进入 `/featsched/ctl`；
    - collector 可以 sudo；
    - 两个 terminal 必须 source 同一个 `runs/current_run_env.sh`。
