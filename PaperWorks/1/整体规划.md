# **论文暂定题目**

## **中文题目**

**计算特征驱动的可编程 CPU 调度：一种基于 eBPF 的应用场景识别与策略自适应框架**

## **英文题目**

**FeatSched: Feature-Driven Programmable Scheduling for Adaptive Application-Aware CPU Resource Management**

---

# **0. 论文核心定位**

## **0.1 本文研究对象**

本文研究的不是某一种特定工作负载，而是更一般的问题：

**能否用可观测、可解释、低开销的计算特征，在线识别应用运行场景，并将其映射到可编程 CPU 调度策略参数，从而在不同场景下自动选择更合适的调度行为？**

## **0.2 本文要证明的核心命题**

本文需要证明：

1. **不同应用场景确实具有可区分的计算特征。**
2. **默认调度器或静态参数无法在所有场景下稳定达到最优或接近最优。**
3. **基于 eBPF/PMU/tracepoint 的低开销特征采集可以支撑在线场景识别。**
4. **“特征 → 场景 → 策略参数”的可解释映射可以带来稳定收益。**
5. **在线策略切换必须受到安全窗、冷却时间、回退机制约束，否则可能产生震荡或性能回退。**

这和 eMFS 中“性能指标比 CPU utilization 更能指导调度”的思路一致，但我们的对象不是 serverless SLO 调度，而是更一般的**计算特征驱动调度**。eMFS 明确指出，调度器设计面临低开销性能采集、及时决策和无需修改内核部署三类挑战，并用 eBPF monitor、资源估计器、SLO-aware 调度器组成反馈闭环，这一点可作为我们设计闭环的参考。

# **1. Introduction：引言大纲**

## **1.1 背景**

现代终端与边缘设备上的应用负载越来越复杂，单一设备上可能同时存在：

|**场景**|**代表负载**|**主要目标**|
|---|---|---|
|交互型任务|UI、音视频、前台服务、ROS 控制链|低延迟、低抖动、少 deadline miss|
|吞吐型任务|编译、压缩、批处理、后台数据处理|高吞吐、短完成时间|
|内存敏感任务|图处理、缓存密集服务、数据扫描|减少 cache miss、内存带宽冲突|
|周期实时任务|控制环、传感器处理、机器人 pipeline|周期稳定性、deadline 可保证|
|混合任务|前台交互 + 后台批处理|前台保护 + 后台资源利用率|

Linux 默认调度器通常基于通用公平性、优先级或 deadline 模型进行调度，但它并不知道应用当前属于什么场景，也不知道任务处于计算密集、内存密集、交互敏感还是周期实时阶段。ALPS 对 CFS 的分析中也指出，CFS 通过 proportional-share 机制隐式控制 CPU 份额，但缺少对单个任务执行顺序和 time slice 的显式控制能力；ALPS 的解决方式是把应用知识通过 eBPF map 传给内核调度逻辑。 

## **1.2 问题陈述**

本文要解决的问题可以表述为：

**如何在不大规模修改 Linux 内核的前提下，构建一个低开销、可解释、可回退的可编程调度框架，使系统能够根据在线计算特征识别当前应用场景，并动态选择调度策略参数？**

## **1.3 关键挑战**

|**挑战**|**具体含义**|
|---|---|
|C1：特征定义难|微架构、调度路径、应用体验指标维度多，容易变成不可解释的黑盒特征堆叠|
|C2：采集开销难控|PMU、tracepoint、eBPF 高频采集可能影响被测负载|
|C3：映射关系不稳定|不同硬件、不同负载强度下，同一特征可能对应不同策略|
|C4：策略切换有风险|过度调整调度参数可能导致震荡、迁移过多、上下文切换增加|
|C5：评估容易跑偏|如果只证明“某个 workload 变快”，就无法支撑“计算特征驱动调度”这个主问题|

## **1.4 预期贡献**

建议把贡献写成 4 点：

1. **提出一套计算特征分层抽象。**  
    覆盖微架构层、调度路径层、系统资源层和应用目标层，并定义适合在线识别的滑动窗口统计形式。
2. **提出一个可解释的“特征—场景—策略参数”映射框架。**  
    不追求端到端黑盒最优，而是用规则、决策树或有限状态机表达场景识别和策略选择逻辑。
3. **实现一个基于 eBPF/PMU/sched_ext 的原型系统。**  
    用户态控制面负责特征聚合与策略生成，内核态 eBPF 数据面负责低开销观测和快速策略执行。ALPS 的 frontend/backend 解耦设计是一个重要参考：其 frontend 在用户态学习策略，backend 通过 eBPF map 在内核中执行策略，避免把复杂策略逻辑全部放进内核。 
4. **通过多场景实验评估有效性、开销、稳定性和可解释性。**  
    实验不只看性能收益，还要包含动机实验、消融实验、敏感性分析、开销分析和异常回退实验。

# **2. Background and Related Work：背景与相关工作**

## **2.1 Linux 调度与可编程调度接口**

这一节介绍：

- CFS / EEVDF / SCHED_FIFO / SCHED_RR / SCHED_DEADLINE；
- cgroup CPU weight / cpu.max / cpuset；
- sched_ext 的基本思想；
- eBPF maps、ring buffer、tracepoint、kprobe、fentry/fexit；
- PMU counters 与 perf event。

UFS 对 sched_ext 和 eBPF 的描述可以作为参考。UFS 指出，sched_ext 允许通过动态加载 eBPF 程序实现调度策略，eBPF map 可以在用户态和内核态之间共享状态；sched_ext hook 可以在任务变为 runnable 或 CPU 需要任务时触发调度逻辑。 

## **2.2 应用感知调度**

相关工作可按方法分类，而不是按 serverless / DB 场景分类：

|**类别**|**代表工作**|**对本文启发**|
|---|---|---|
|应用感知 OS 调度|ALPS、UFS、SFS、ghOSt、Syrup|应用知识如何进入调度器|
|eBPF 增强内核机制|ALPS、UFS、FetchBPF、BMC、Prism|eBPF 作为低侵入扩展路径|
|性能反馈驱动资源管理|eMFS、Cilantro、Quasar|不只看 CPU utilization，而看应用目标|
|微秒级/低尾延迟调度|Shinjuku、Caladan、Shenango、Concord|低延迟场景中的抢占、核心分配、反馈控制|
|特征工程与瓶颈诊断|Top-Down、PMU profiling、perf sched|特征选择和瓶颈解释|

## **2.3 本文与相关工作的区别**

建议明确写：

本文不针对某一类固定 workload 设计专用调度器，而是研究一种通用的计算特征驱动方法。与 ALPS 关注 serverless 函数排序不同，本文关注跨场景特征识别和策略参数选择；与 UFS 关注数据库前后台任务隔离不同，本文关注应用语义和运行时特征如何泛化为策略映射；与 eMFS 关注 SLO-aware FaaS 调度不同，本文关注更一般的“计算特征—策略”闭环。

# **3. Problem Definition：要研究的问题**

## **3.1 系统模型**

实验一建议限定为 CPU 单子系统：

```text
输入：
  多源运行时特征 F(t)
  应用场景标签或隐含目标 G
  当前系统状态 S(t)

输出：
  CPU 调度策略参数 P(t)

目标：
  在满足稳定性约束 C 的条件下，优化应用目标 O
```

其中：

### **运行时特征 F(t)**

|**层次**|**特征**|
|---|---|
|微架构层|IPC、LLC miss rate、branch miss rate、cycles、instructions|
|调度路径层|runqueue length、runqueue latency、wakeup latency、context switch rate、migration rate|
|CPU 使用层|per-core utilization、steal/idle time、cgroup CPU usage|
|应用层|p95/p99 latency、jitter、deadline miss、throughput、task completion time|

### **策略参数 P(t)**

|**参数**|**含义**|
|---|---|
|queue placement|任务进入哪个 DSQ / priority queue|
|time quantum|每类任务的执行粒度|
|virtual runtime adjustment|调整任务排序|
|cgroup weight|分配 CPU 份额|
|CPU affinity|是否绑定到特定 core / NUMA node|
|migration threshold|何时允许跨核迁移|
|preemption rule|何时抢占后台任务|
|fallback mode|策略异常时回退 CFS/EEVDF/默认策略|

## **3.2 Research Questions**

建议论文正式写成 5 个 RQ：

### **RQ1：计算特征是否足以区分不同应用场景？**

例如：CPU-bound、memory-sensitive、latency-sensitive、periodic real-time、mixed foreground/background 是否在 IPC、LLC miss、runqueue latency、context switch、deadline miss 等维度上呈现稳定差异？

### **RQ2：静态调度策略是否无法在多场景下稳定最优？**

需要证明：CFS/EEVDF、SCHED_FIFO/RR、SCHED_DEADLINE、静态 cgroup、固定 sched_ext 策略各有适用场景，但没有一个策略在所有场景下都稳定最优。

### **RQ3：能否建立可解释的“特征—场景—策略参数”映射？**

目标不是训练一个黑盒模型，而是形成可检查、可调试、可解释的规则：

```text
if runqueue_latency high and IPC high and LLC_miss low:
    scene = CPU_contention_latency_sensitive
    policy = shorter quantum + foreground priority + migration restriction

if LLC_miss high and migration_rate high:
    scene = memory/cache_sensitive
    policy = stronger affinity + reduced migration + cache-friendly placement

if deadline_miss increasing and utilization moderate:
    scene = scheduling_path_problem
    policy = priority/queue boost + preemption adjustment
```

### **RQ4：动态映射是否优于静态策略？**

需要证明在 workload phase change 下，动态策略能快速收敛，并且不会因策略切换造成明显震荡。

### **RQ5：系统开销和安全性是否可接受？**

需要量化 eBPF hook、PMU 采样、map update、用户态控制循环、策略执行等开销，并证明异常时可以回退。

# **4. Motivation：需要证明的结论**

Motivation 部分建议写成 4 个小节，每节最后给一个 Takeaway。三篇文章中，eMFS 的 Takeaway 写法最适合借鉴；UFS 的 failure mode 分析也适合借鉴。

---

## **4.1 Motivation 1：单一系统指标不足以反映应用目标**

### **要证明的结论**

**CPU utilization、平均 CPU share 等 proxy metric 不能充分解释应用延迟、抖动和 deadline miss；调度策略需要引入更接近应用目标的特征。**

### **实验设计**

构造 2-3 类负载：

|**场景**|**负载组合**|**预期现象**|
|---|---|---|
|A|latency-sensitive foreground + CPU batch background|CPU 利用率接近，但 p99 latency 差异大|
|B|periodic task + memory bandwidth workload|CPU 利用率不高，但 deadline miss 增加|
|C|interactive workload + frequent wakeup task|平均吞吐稳定，但 wakeup latency 和 jitter 恶化|

### **推荐图表**

- Fig. 1(a)：CPU utilization vs p99 latency；
- Fig. 1(b)：CPU utilization vs deadline miss；
- Fig. 1(c)：runqueue latency timeline。

### **借鉴点**

eMFS 明确指出 CPU utilization 难以反映 end-to-end latency 或 throughput，并主张使用真实性能指标进行调度；其 motivation 中还对 queuing delay 和 CPU usage 的不一致做了专门分析。

## **4.2 Motivation 2：不同场景需要不同调度策略**

### **要证明的结论**

**不存在一个静态策略在所有计算特征场景下都最优；调度策略应随场景切换。**

### **实验设计**

使用相同硬件，比较：

|**Baseline**|**用途**|
|---|---|
|CFS / EEVDF|默认公平调度|
|SCHED_FIFO|强优先级抢占|
|SCHED_RR|固定 time quantum|
|SCHED_DEADLINE|周期实时任务 baseline|
|Static cgroup|静态资源权重|
|Static sched_ext policy A|偏低延迟|
|Static sched_ext policy B|偏吞吐|
|Best-of-N oracle|离线选择每个场景最优策略|

观察不同场景下的 winner 是否不同。

### **推荐图表**

- Fig. 2：不同 workload 场景下各策略的 normalized score；
- Table 1：各策略适用场景与失败模式。

### **借鉴点**

UFS 在 motivation 中不是简单说“默认调度不好”，而是分解不同调度器的 failure mode：EEVDF 在高低优先级混合时 placement 不佳，FIFO/RR 在同优先级 CPU-bursty 与 CPU-bound 混合时失败。这个“不同策略不同失败模式”的论证方式非常适合我们。

## **4.3 Motivation 3：计算特征可以预测“该用哪类策略”**

### **要证明的结论**

**微架构特征、调度路径特征和应用目标特征联合起来，可以形成可解释的场景簇，并与策略选择相关。**

### **实验设计**

收集不同 workload 的特征窗口：

```text
window = 100ms / 500ms / 1s
features = {
  IPC, LLC_miss, branch_miss,
  runqueue_latency, wakeup_latency,
  context_switch_rate, migration_rate,
  p95_latency, deadline_miss, throughput
}
label = best_policy_under_this_window
```

做三类分析：

|**分析**|**目的**|
|---|---|
|PCA / t-SNE / UMAP 可视化|看场景是否可分|
|决策树 / rule list|得到可解释规则|
|feature importance|确定哪些特征最关键|

## **4.4 Motivation 4：在线策略切换需要安全机制**

### **要证明的结论**

**没有 safety guard 的动态策略可能带来震荡、过度迁移、上下文切换增加或 deadline miss 突增；因此本文必须引入安全窗和回退机制。**

### **实验设计**

构造 phase-changing workload：

```text
Phase 1: foreground latency-sensitive
Phase 2: background CPU-bound 增强
Phase 3: memory/cache-sensitive workload 加入
Phase 4: periodic real-time workload 加入
Phase 5: 回到平稳状态
```

比较：

|**系统**|**说明**|
|---|---|
|Static best policy for Phase 1|证明静态策略在其他阶段失效|
|Adaptive without guard|证明动态策略可能震荡|
|Adaptive with guard|完整系统|
|Oracle|每个阶段离线最优|

# **5. System Design：系统设计思路**

建议系统命名为 **FeatSched**，整体采用：

```text
eBPF/PMU Data Plane
        ↓
Feature Aggregator
        ↓
Scenario Identifier
        ↓
Policy Mapper
        ↓
Safety Guard
        ↓
sched_ext / cgroup / affinity Policy Executor
        ↓
Feedback Monitor
```

UFS 的 application hinting 说明 eBPF map 可以作为用户态应用和内核调度器之间的通信通道；用户态写入 hint，内核态 eBPF 程序读取并影响调度决策。  我们可以把这个思想泛化为：不是只传数据库锁，而是传**场景标签、任务类型、策略参数和安全状态**。

---

## **5.1 设计目标**

|**目标**|**含义**|
|---|---|
|G1 Low-overhead observability|用 eBPF/PMU/tracepoint 低开销采集多层特征|
|G2 Explainable mapping|用可解释规则把特征映射到场景和策略参数|
|G3 Programmable enforcement|通过 sched_ext / cgroup / affinity 执行策略|
|G4 Online adaptivity|支持 workload phase change 下的在线调整|
|G5 Safety and fallback|策略越界时快速回退默认策略|
|G6 Portability|原型先支持 x86 Linux，后续扩展 ARM big.LITTLE|

---

## **5.2 模块一：Feature Data Plane**

### **功能**

负责采集低层特征。

### **数据来源**

|**数据源**|**采集内容**|
|---|---|
|PMU / perf event|IPC、LLC miss、branch miss、cycles、instructions|
|sched tracepoint|sched_switch、sched_wakeup、runqueue latency、context switch|
|eBPF maps/ringbuf|per-task/per-cgroup 聚合数据|
|cgroup stats|CPU usage、throttling、cpu pressure|
|application probe|latency、jitter、deadline miss、throughput|

### **设计原则**

- 高频事件在内核态预聚合；
- 用户态只读取窗口统计量；
- 所有事件带单调时间戳；
- 按 task / thread group / cgroup 三层索引；
- 高开销特征按需开启。

---

## **5.3 模块二：Feature Aggregator**

### **功能**

将原始事件转换为稳定特征向量。

```text
raw events → sliding window → normalized features → feature vector
```

### **窗口统计**

|**统计量**|**用途**|
|---|---|
|mean|平均趋势|
|p95 / p99|尾部行为|
|variance|稳定性|
|rate|事件频度|
|slope|是否正在恶化|
|burstiness|是否有突发|

---

## **5.4 模块三：Scenario Identifier**

### **功能**

识别当前 workload 场景。

### **初始场景集合**

|**场景标签**|**典型特征**|
|---|---|
|CPU-bound throughput|IPC 高，runqueue 长，latency 不敏感|
|Latency-sensitive interactive|wakeup 频繁，runqueue latency 高，p99 latency 重要|
|Periodic real-time|周期稳定，deadline miss 是核心指标|
|Memory/cache-sensitive|LLC miss 高，migration 后性能波动|
|Mixed foreground/background|前台 p99 受后台影响|
|Overloaded/unstable|deadline miss、context switch、migration 同时升高|

### **模型选择**

建议先用：

1. rule-based FSM；
2. decision tree；
3. lightweight classifier 作为对照。

不要第一版就上复杂深度模型。实验一的核心是**可解释映射**，不是模型精度竞赛。

---

## **5.5 模块四：Policy Mapper**

### **功能**

把场景映射为策略参数。

示例：

|**场景**|**策略参数**|
|---|---|
|CPU-bound throughput|增大 time quantum，减少抢占，降低迁移频率|
|Latency-sensitive interactive|缩短前台排队，提升优先级，允许抢占后台|
|Periodic real-time|使用 deadline/budget 保护，限制后台干扰|
|Memory/cache-sensitive|增强 affinity，减少跨核迁移，避免 cache 破坏|
|Mixed foreground/background|前台优先，后台 idle / low-share 执行|
|Overloaded|回退保守策略，降低策略切换频率|

策略执行可以分两级：

|**层次**|**第一版实现**|
|---|---|
|coarse-grained|cgroup weight、cpuset、nice、SCHED policy|
|fine-grained|sched_ext DSQ、time quantum、vruntime ordering、dispatch rule|

---

## **5.6 模块五：Policy Executor**

### **可选实现路径**

|**路径**|**优点**|**风险**|
|---|---|---|
|cgroup + cpuset + nice|易实现、风险低|粒度粗|
|sched_ext|可编程性强、可动态加载|需要较新内核|
|修改 CFS hook|控制力强|工程侵入大|
|用户态调度器|开发方便|通信开销和延迟较高|

实验一建议采用：

**cgroup/cpuset 作为 fallback，sched_ext 作为主要执行器。**

UFS 的实现路径值得借鉴：它基于 sched_ext/eBPF 实现调度策略，不需要修改 Linux 源码；eBPF map 用于承载用户态传来的应用语义。 

---

## **5.7 模块六：Safety Guard and Fallback**

### **必须有**

|**机制**|**作用**|
|---|---|
|policy legality check|策略参数不能越界|
|cooldown window|避免频繁切换|
|hysteresis|避免边界震荡|
|health score|判断系统是否恶化|
|fallback policy|恢复 CFS/EEVDF/static-safe|
|rollback log|可复现、可定位问题|

### **回退条件示例**

```text
if deadline_miss_rate > threshold for N windows:
    rollback_to_safe_policy()

if migration_rate increases by 3x and latency not improved:
    disable_affinity_adjustment()

if eBPF event loss > threshold:
    reduce sampling rate or disable adaptive mode()
```

# **13. 一句话版本**

这篇论文可以这样概括：

本文提出 FeatSched，一个基于 eBPF/PMU/sched_ext 的计算特征驱动可编程 CPU 调度框架。它通过低开销采集微架构、调度路径和应用目标特征，在线识别应用场景，并将场景映射为可解释的调度策略参数。实验表明，单一静态调度策略无法适配多样化计算场景，而 FeatSched 能在 phase-changing workloads 中稳定降低尾延迟、deadline miss 和调度路径开销，同时保持可控的监控与策略切换开销。
